Data-intensive Applications

Data-intensive computing is a class of parallel computing applications which use a data parallel approach to process large volumes of data typically terabytes or petabytes in size and typically referred to as big data. Data intensive applications exhibit the following novel features compared with a typical computing application:

  • I/O: Intensive computing time on I/O & data manipulation
  • ACID: Atomicity, Consistency, Isolation, Durability
  • Scalability: Vertical (scaling-up) vs. horizontal (scaling-out) scaling

Challenges (Scalability Focused)

  • Load Balancer

    Fan out the tremendous amount of incoming requests to meet the SOL.

  • Resource Provision

    Scaling out to meet the SOL by over provisioning.

  • Performance Modeling

    To understand/estimate the distributed system at scaling.

Chapter 3: USBA Model

Motivations

Requirements on extending our existing IPSO model to accommodate the following features:

  • Multi-stage: Data-intensive applications tends to include multiple stages per-execution;
  • Predictable: If it's a prediction model then it can be very useful for resource provisioning;
  • Simple: We want it to be simple so that it can be easily adopted.

Unified Scaling Models

Amdahl/Gustafson's model

System Model for Amdahl's and Gustafson's laws. Each black dot represents a base task and each white box represents an executor. It has two stages, i.e., a serial and a parallel execution stage.

Unified Scaling Models (cont'd)

Unified Scaling models for Big Data (USBA)

A system model with discretized workload. Stages are divided by data transformations that potentially involve large sequential overhead caused by the barrier, normally known as shuffle boundaries. Each white circle represents a driver process in that stage which takes input and dispatches tasks to its workers. The workers are usually running VM instances on core/task nodes of the cluster, depicted as white squire boxes in the figure. And each black circle represents a task being assigned to the worker. Also, when a multi-stage workload contains multiple chains of stages, the longest chain of stages with dependencies shall drive its performance in terms of execution latency.

Chapter 4: Pigeon Scheduler Project

Challenges

Today's warehouse level data centers are facing novel challenges on job scheduling:

  • Clusters are growing larger (tens of thousands nodes/workers);
  • A data-intensive application's job typically involves a great number of tasks;
  • The mixture of heterogeneous workloads (short & long jobs) triggers many issues.

Before Pigeon Was Invited...

Centralized job scheduling

A scheduler manages all the workers’ resources in a cluster

Distributed scheduling (Sparrow)

A scheduler needs to maintain all probes (inefficient!).

Hybrid scheduling (Eagle, Hawk)

Resources are reserved and jobs are separately scheduled, but has "head-of-line" blocking issue.

We introduce the Pigeon scheduler!



Fast

Pigeon is designed as a new architecture, hierarchical job scheduler that guarantees lighting-fast job scheduling in its frontend.

Hierarchical

In Pigeon, a master level is introduced for task distribution and heterogeneous workloads are fully handled.

Scalable

Pigeon fully solves the scalability issues by offering both benefits from centralized and distributed schedulers.

Open-sourced

We release both Pigeon cloud engine and Pigeon Spark plugin as open source projects under MIT license.

What makes Pigeon great?

For it has solved some key challenges in existing schedulers:

1) Scalability

Imagine a cluster of 100 nodes, the number of master can be only 1% of total workers, this reduces 99% status maintenance cost

Pigeon can greatly reduce status maintenance costs in job schedulers

2) Efficiency

Imagine a cluster of 100 nodes that runs at 90% load, the probability of a task finding an idle worker in a group will be 1-0.9100 =99.99734

Pigeon removes head-of-line blocking and has statistical multiplexing gain within a group

Pigeon Implementation

The Pigeon open source project consists several parts:

Chapter 5: Conclusion

Summary

In this dissertation, we present a new performance model and job scheduler for today's data centers:

The first presented modeling approach (IPSO) in this dissertation is inspired by the classic Amdahl's model which can be applied in a wide domain of parallel computing. This new model generalizes the classic performance models of its predecessors and renders them as its special cases. IPSO can meet both analytical and accurate system performance modeling requirements.

The second model (USBA) is introduced in consideration of today's data-intensive applications' multi-stage feature. It can also be used a prediction model given the proper training data. At the end of this work and the previous one, we release our modeling tool as an open source project.

Move on towards the investigation of resource provision techniques in modern data centers, we allocate several important factors that can greatly impact the performance of execution such as head-of-line blocking, long job starvation, etc. And based on these observations, we propose the Pigeon, the distributed, hierarchical scheduler for today's data centers.

In our next steps, we plan to transplant the Pigeon cloud engine into today's edge computing environment.
